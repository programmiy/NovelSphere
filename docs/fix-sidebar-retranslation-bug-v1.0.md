# 버그 수정: 사이드바 중복 번역 문제 해결 (v1.0)

## 변경 사항 요약
콘텐츠 스크립트(`content.js`)의 초기화 로직을 리팩토링하여, 페이지 로드 시 사이드바가 이미 데이터베이스에 저장된 번역 기록을 인식하지 못하고 다시 번역을 시도하던 문제를 해결했습니다.

## 문제 원인
기존 로직은 "페이지에 적용" 기능이 활성화된 경우에만 서버로부터 기존 번역 기록을 가져왔습니다. 이 기능이 비활성화된 상태에서는, 콘텐츠 스크립트가 페이지 내에 어떤 번역 기록이 저장되어 있는지 전혀 알지 못한 채, 페이지에서 발견된 모든 일본어 문장을 새로 번역하려고 시도했습니다.

또한, 백엔드의 번역 API(`POST /translate`)는 `url`을 필수로 요구하도록 변경되었으나, 백그라운드 스크립트(`background.js`)에서 이 값을 전송하지 않아 번역 요청이 실패하는 문제도 함께 발견되었습니다.

## 상세 수정 내용

### 1. `content.js` 초기화 로직 개선
-   **항상 번역 기록 먼저 조회**: `init` 함수를 수정하여, 페이지가 로드될 때 항상 서버에 현재 URL에 대한 기존 번역 기록을 요청하고, 그 결과를 `translationCache`에 저장하도록 변경했습니다.
-   **선 조회, 후 적용**: 이제 스크립트는 먼저 캐시를 채운 뒤, 별도로 `chrome.storage`를 확인하여 "페이지에 적용" 플래그가 켜져 있을 경우에만 캐시된 내용을 페이지 본문에 적용합니다.
-   **중복 번역 방지**: 새로운 번역을 찾는 `translateAllPTags` 함수가 DOM 대신 `translationCache`를 직접 확인(`!translationCache.has(p.id)`)하도록 수정하여, 이미 캐시에 있는(즉, DB에 저장된) 문장은 번역 대상에서 제외되도록 했습니다.
-   **단순화된 이벤트 리스너**: `chrome.storage.onChanged` 리스너가 "페이지에 적용" 상태 변경 시, 복잡한 로직 대신 페이지를 새로고침하도록 단순화했습니다. 새로고침된 페이지는 개선된 `init` 로직에 따라 상태를 올바르게 반영합니다.
-   **클라이언트 측 제외 로직 제거**: 서버에서 번역 제외 처리를 모두 담당하게 되면서, `content.js` 내의 불필요한 제외 목록 관련 코드를 제거했습니다.

### 2. `background.js` API 호출 수정
-   `handleBatchTranslate` 함수가 서버의 `/translate` 엔드포인트로 요청을 보낼 때, 요청 본문(`body`)에 현재 탭의 URL을 `url` 필드에 담아 전송하도록 수정했습니다. 이로써 백엔드가 URL별 제외 목록을 올바르게 확인할 수 있게 되었습니다.

## 기대 효과
-   **버그 해결**: 더 이상 저장된 번역을 중복으로 다시 번역하지 않아 불필요한 API 호출과 사용자 혼란을 방지합니다.
-   **성능 개선**: 페이지 로드 시 데이터 흐름을 명확히 하여 성능과 안정성을 개선했습니다.
-   **코드 품질 향상**: 초기화 로직을 단순화하고 단일 책임 원칙에 가깝게 리팩토링하여 코드의 가독성과 유지보수성을 높였습니다.
