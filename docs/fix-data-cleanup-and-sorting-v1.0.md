# 기능 추가 및 버그 수정: 데이터 정리 및 PID 정렬 개선 (v1.0)

## 변경 사항 요약
1.  특정 패턴(la/lp + 숫자)의 PID를 가진 기존 번역 기록을 데이터베이스에서 일괄 제외 처리하는 관리자용 엔드포인트를 추가했습니다.
2.  번역 기록 페이지에서 PID가 올바르게 정렬되지 않던 버그를 수정했습니다.

## 1. 특정 PID 일괄 제외 기능

### 변경 내용
-   **백엔드 (`server/server.py`)**: 
    -   `POST /admin/cleanup_pids` 라는 새로운 관리자용 엔드포인트를 추가했습니다.
    -   이 엔드포인트는 호출 시, 데이터베이스의 `translations` 테이블에서 `pid`가 `la` 또는 `lp`로 시작하고 뒤에 숫자가 오는 모든 기록을 조회합니다.
    -   조회된 기록들의 `(url, original)` 정보를 `excluded_sentences` 테이블에 추가하여, 향후 번역에서 영구적으로 제외되도록 합니다.
    -   `excluded_sentences` 테이블에 추가한 후, `translations` 테이블에서 해당 기록들을 완전히 삭제합니다.

### 기대 효과
-   **데이터 정합성**: 번역이 불필요한 특정 패턴의 데이터를 일괄적으로 정리하고, 향후 번역 대상에서도 제외 처리하여 데이터의 정합성을 높입니다.
-   **관리 편의성**: 수동으로 데이터를 삭제할 필요 없이, 엔드포인트 호출 한 번으로 데이터베이스를 정리할 수 있어 관리 편의성이 증대됩니다.

---

## 2. 번역 기록 페이지 PID 정렬 버그 수정

### 문제 원인
- **서버**: 번역 기록을 가져오는 `/translations_by_folder` 엔드포인트가 데이터를 `timestamp DESC`(최신순)으로 정렬하고 있었습니다. 긴 문장과 일반 문장은 처리 시간 차이로 `timestamp`가 달라, 이 정렬 방식은 PID 순서를 완전히 무시하는 결과를 초래했습니다.
- **클라이언트**: 이전의 PID 정렬 로직이 `la`와 같은 다양한 PID 접두사를 처리하지 못하는 문제가 있었습니다.

### 변경 내용
-   **백엔드 (`server/server.py`)**:
    -   `/translations_by_folder` 엔드포인트의 SQL 쿼리에서 `timestamp DESC` 정렬을 제거하여, 클라이언트가 PID 정렬을 온전히 제어할 수 있도록 수정했습니다.
-   **프론트엔드 (`extension/history.js`)**:
    -   번역 기록 페이지(`history.html`)의 정렬 로직을 `String.prototype.localeCompare()` 메소드와 `{ numeric: true }` 옵션을 사용하는 방식으로 변경했습니다.
    -   이 방식은 `p1`, `p10`, `la2` 등 문자와 숫자가 혼합된 다양한 형식의 PID를 자연 정렬(Natural Sort)하여 사용자의 직관에 맞는 정확한 순서를 보장합니다.

### 기대 효과
-   **정확한 정렬**: 이제 번역 기록 페이지에서 모든 번역 항목들이 PID 순서에 따라 일관되고 정확하게 정렬됩니다.
-   **안정성 향상**: 서버와 클라이언트의 역할을 명확히 분리하고, 클라이언트에서 더 안정적인 최신 정렬 API를 사용하여 코드의 안정성과 예측 가능성을 높였습니다.
