# 버그 수정: 번역 제외 기능 로직 개선 (v1.0)

## 변경 사항 요약
번역에서 제외된 문장이 번역 기록에서는 삭제되지만, 페이지에 다시 방문했을 때 빈 번역문으로 다시 번역을 시도하던 문제를 해결했습니다. 이제 제외된 문장은 번역 대상 자체로 인식되지 않습니다.

## 문제 원인
기존의 제외 기능은 서버에서만 제외 목록을 관리하고 있었습니다. 콘텐츠 스크립트(`content.js`)는 이 제외 목록에 대한 정보 없이, 페이지에 있는 모든 일본어 텍스트에 대해 번역을 요청했습니다. 서버는 제외된 텍스트에 대해 빈 문자열을 반환했고, 이 빈 문자열이 사용자에게 빈 번역문으로 보였습니다. 사용자가 원한 것은 번역 시도 자체를 안 하는 것이었습니다.

## 상세 수정 내용

### 1. 백엔드 (`server/server.py`)
-   **`GET /excluded_sentences` 엔드포인트 추가**:
    -   URL을 쿼리 파라미터로 받아, 해당 URL에 대해 제외된 모든 원문(`original`) 텍스트의 목록을 JSON 형태로 반환하는 새로운 API 엔드포인트를 구현했습니다.

### 2. 프론트엔드 (`extension/content.js`)
-   **제외 목록 사전 로드 (`init` 함수)**:
    -   페이지가 로드될 때 실행되는 `init` 함수를 수정하여, 기존 번역 기록을 가져오는 것과 동시에 새로 추가된 `/excluded_sentences` 엔드포인트를 호출하도록 했습니다.
    -   `Promise.all`을 사용하여 번역 기록과 제외 목록을 병렬로 효율적으로 가져옵니다.
    -   가져온 제외 목록은 `exclusionSet`이라는 전역 `Set` 객체에 저장하여, 이후의 조회 성능을 최적화했습니다.

-   **번역 시도 전 제외 목록 확인 (`translateAllPTags` 함수)**:
    -   새로운 번역 대상을 수집하는 `translateAllPTags` 함수에서, 번역할 텍스트를 큐에 추가하기 전에 `exclusionSet.has(text)`를 통해 해당 텍스트가 제외 목록에 있는지 먼저 확인하도록 로직을 추가했습니다.
    -   만약 텍스트가 제외 목록에 포함되어 있다면, 번역을 요청하지 않고 즉시 건너뜁니다.

## 기대 효과
-   **완벽한 제외 기능**: 이제 사용자가 한 번 제외한 문장은 더 이상 번역 대상으로 인식되지 않으며, 빈 번역문이 나타나는 현상이 완전히 해결되었습니다.
-   **성능 개선**: 불필요한 번역 API 요청을 원천적으로 차단하여 API 할당량을 절약하고, 클라이언트 측의 처리 부담을 줄였습니다.
-   **올바른 아키텍처**: 콘텐츠 스크립트가 번역에 필요한 모든 정보(기존 번역, 제외 목록)를 사전에 인지하고 동작하도록 아키텍처를 개선하여, 코드의 예측 가능성과 안정성을 높였습니다.
