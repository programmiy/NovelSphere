# 자동 번역 프로세스 중지 로직 변경

## 개요
자동 번역 프로세스의 안정성을 높이고 API 사용량을 관리하기 위해, 특정 조건 하에 번역 요청을 제어하는 새로운 로직이 도입되었습니다. 이 로직은 크게 두 가지 시나리오를 다룹니다.

## 주요 변경 사항

### 1. '경계 시간' 기반 자동 중지 (오후 5시)

**목적:** API 사용량 리셋 시간(오후 5시 추정) 전에, 장시간 실행되던 자동 번역 프로세스를 자동으로 중지시켜 일일 사용량을 분석할 시간을 확보합니다.

**동작 방식:**
- **오후 5시 이전에 시작된** 자동 번역 프로세스가 계속 실행되다가, 오후 5시를 넘어서 다음 번역을 요청하는 경우, 해당 요청은 **1회에 한해 거부**됩니다.
- 거부 메시지: `"오후 5시가 되어 자동 번역을 중지합니다. 새로고침 후 다시 시도해주세요."`
- 이 거부 동작 이후, 내부 상태가 초기화됩니다. 따라서 사용자가 **새로고침 하거나 새로 번역을 시작**하는 경우, 오후 5시가 넘었더라도 정상적으로 번역이 실행됩니다.
- 즉, 이 기능은 **오후 5시 이후에 새로 시작된 번역은 막지 않으며**, 오직 5시 이전에 시작되어 계속 실행되던 **자동 번역 프로세스만**을 중지 대상으로 합니다.

### 2. 연속적인 503 에러 처리

**목적:** API 서버의 일시적인 과부하(503 Service Unavailable)에 대해, 불필요한 재시도를 줄이고 현재 요청을 안정적으로 실패 처리합니다.

**동작 방식:**
- 단일 번역 요청(하나의 번역 묶음)에 대한 재시도 중 **연속 3회** 503 에러가 발생하면, 해당 요청은 즉시 실패 처리됩니다.
- 거부 메시지: `"반복적인 API 과부하로 현재 요청을 처리할 수 없습니다. 잠시 후 다시 시도해주세요."`
- 이 로직은 **현재 번역 요청에만** 적용되며, 서버의 다른 기능에 영향을 주지 않습니다. 잠시 후 새로고침하여 다시 시도할 수 있습니다.

## 기술적 구현
- `server/routers/translation.py` 파일이 수정되었습니다.
- '경계 시간' 로직을 위해, 오후 5시 이전에 요청이 시작되었는지를 추적하는 서버 측 전역 플래그(`process_started_before_5pm`)가 도입되었습니다.
- `tenacity`를 사용한 재시도 로직이 수동 재시도 루프로 변경되어, 503 에러 카운팅을 보다 정밀하게 제어합니다.
