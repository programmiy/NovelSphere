앞으로 모든 답변은 한국어로 하고, 전문가처럼 간결하게 핵심만 말해줘.

[코드를 수정해야할 때]

1. **코드 수정:** 요청된 코드 수정 작업을 수행합니다. 복잡한 작업일경우 분할해 수행한다.
2. 기능이 정상작동되는지 확인하고 이후의 단계를 진행합니다.
3. **[분기] 문서화 필요 여부 확인** (기존 6번을 3번으로 이동)

   * **IF (요청이 단순 오타 수정, 변수명 변경, 주석 추가 등 경미한 작업일 경우):**
     * `3-1. 문서 생성:`
       * 비슷한 수정사항이 있는 문서가 있는지 `docs/`폴더를 확인합니다.
       * 확인한 정보를 바탕으로 아래 단계를 수행합니다.
       * 수정 사항에 대한 마크다운 문서를 `docs/` 폴더 안에 생성합니다.
       * 파일명은 `docs/[기능]-[YYYY-MM-DD].md` 형식으로 합니다.
       * (예: `docs/fix-login-bug-2025-10-25.md`)
       * (만약 같은 날 같은 기능의 문서가 또 생기면, `-2`를 붙여 구분합니다.)
     * `4. 커밋` 단계로 이동합니다.
   * **ELSE (신규 기능 추가, 로직 변경 등 대부분의 작업일 경우):**
     * `3-1. 문서 생성:`

       * 비슷한 수정사항이 있는 문서가 있는지 `docs/`폴더를 확인합니다.
       * 확인한 정보를 바탕으로 아래 단계를 수행합니다.
       * 수정 사항에 대한 마크다운 문서를 `docs/` 폴더 안에 생성합니다.
       * 파일명은 `docs/[기능]-[YYYY-MM-DD].md` 형식으로 합니다.
       * (예: `docs/fix-login-bug-2025-10-25.md`)
       * (만약 같은 날 같은 기능의 문서가 또 생기면, `-2`를 붙여 구분합니다.)
     * `4. 커밋` 단계로 이동합니다.
4. **커밋:**

   * (IF 경미한 작업) 수정된 **코드 파일**만 스테이징합니다.
   * (ELSE 대부분의 작업) 수정된 **코드 파일**과 생성된 **문서 파일**을 함께 스테이징합니다.
   * 파일들을 **하나의 커밋**으로 묶습니다.
5. 원격 저장소 는 `main`만 존재 , `origin`사용하지 않음
6. [분기] 푸시 여부 확인-푸시 여부를 질문합니다-

   * **IF (질문결과 : 푸시 승인):**
     * ```shell
       git status
       ```

       를 통해 스테이징되지 않은 파일들을 추가로 파악후, 스테이징되지 않은 파일이 있다면 해당 파일의 변경사항도 작성했던 문서에 작성
     * `7. **마무리**` 단계로 바로 이동합니다.
   * **ELSE (질문결과 : 푸시 미승인):**
     * 사용자의 추가 명령을 기다립니다.
     * 필요에따라 1단계부터 다시 수행합니다.
7. **마무리:** (git push main extra:main (로컬 extra를 원격 EXTRA에 푸시)
8. 자동화된 병합 워크플로우 (MCP/GitHub Actions)

   1. **main pull 이후 작업 브랜치로 복귀**

      * `main` 브랜치가 최신화되었으므로, 로컬 `extra` 브랜치도 최신 `main`을 반영합니다.
        ```shell
        git checkout main
        git pull main main
        git checkout extra
        git rebase main
        ```
